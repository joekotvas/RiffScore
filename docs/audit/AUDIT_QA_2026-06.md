## Overall Trust Verdict

**Act on these docs — with three named carve-outs.** Both documents are substantially reliable. **False-positive profile among the ~109 findings.** Counting at the finding level across all 14 clusters:
- **Validated (correct as stated, severity agreed): ~70%.** The vast majority. Many are crisply located with the right severity.
- **Nuance (true but mis-scoped, framing imprecise, or live-vs-latent boundary blurred): ~20%.** Real defects whose blast radius or trigger conditions are narrower than the prose implies (typically: the bug needs the public `makeTuplet`/`addNote` API or a forced out-of-key note, not the normal UI path).
- **Overstated severity: ~6%.** Real defects rated one notch too high (e.g. `tenor-clef-pitch-offset-wrong` rated critical but subsumed; `no-tuplet-aware-beaming` major for a visual-only issue).

**The single biggest structural defect is double-counting, not wrongness.** The "109 findings" total is inflated by genuine duplicates that should be merged (detailed in the table). The real distinct-defect count is meaningfully lower. This matters because it skews the remediation tracker's effort/severity weighting.

**Remediation plan aim:** sound in direction, with two material under-acknowledged couplings (Phase 1C serialization/schemaVersion; Phase 1A delete-vs-repurpose) and one product mis-sequencing (export fidelity ahead of interactive/API correctness). None invalidate the plan; all are fixable framing edits.

## Design Challenges (by lens, high-severity first)

### Architecture lens

**[HIGH] Phase 1C quant-base migration silently rescales serialized chord positions.** *Verdict: flag.* The plan rescales the internal quant grid (×105) but `ChordSymbol.quant` is a **serialized, measure-local integer** (`types.ts:64-72`, emitted verbatim by `jsonExporter.ts`). Multiplying the base re-interprets every saved chord's position, and there is **no `schemaVersion`** anywhere (verified: 0 grep hits) — so migration cannot even tell base-16 from base-1680 data. Phase 1C therefore depends on the schemaVersion work the plan parks in Phase 4 (an ordering inversion). **Recommendation:** pull `schemaVersion` forward as a prerequisite of 1C; the 1C migration must rescale **all** `chordTrack.quant`, not just tuplet-anchored ones; add a load→render round-trip fixture test.

**[HIGH/MEDIUM] The ×105 multiplier is mathematically wrong.** *Verdict: validated defect, flawed fix.* Verified in `tonal`: `1*105*1.5 = 157.5` (a dotted-64th is non-integer on the ×105 grid because 105 is odd). The minimum correct factor is `2*LCM(3,5,7)=210` (verified integer). The docs are internally inconsistent (plan says 105, one finding says 840, another 2520). **Recommendation:** use a named `TICKS_PER_QUARTER` of ≥210 (or exact `{num,den}` rationals), state that a fixed grid is **not** general to nested tuplets (verified: 3:2-of-3:2 = 46.67) — acceptable only because nesting is rejected at `entry.ts` — and add a startup invariant test asserting every supported ratio×base is integer.

**[MEDIUM] `note.accidental`: repurpose, do not delete.** *Verdict: nuance.* The audit's premise that "render/audio read `note.accidental`" is partly wrong — verified that `useAccidentalContext.ts:39` already derives the on-screen glyph from `note.pitch`; the field is read mainly by exporters and the spacing lookahead (it is vestigial for rendering). More importantly, the field is the **only** place a courtesy/forced/editorial accidental intent can live — pitch alone cannot encode "show this natural even though the key already makes it natural." Deleting it forecloses that. **Recommendation:** repurpose into an explicit optional display-policy override (e.g. `accidentalDisplay?: 'force'|'courtesy'|'hide'|null`) that the resolver consumes at highest priority but that **never** affects sounding pitch; keep a load-time migration that reconciles legacy values into the pitch for one deprecation cycle.

**[MEDIUM] Validation placement contradiction in commands.** *Verdict: nuance.* The plan says "commands stay dumb, enforce at the API surface" (`no-capacity-validation`) but elsewhere says capacity "should be enforced at the command/model layer (`AddEventCommand`)" (`midi-entry`). These contradict, and `ScoreEngine.dispatch` swallows exceptions into a boolean — so a command that "rejects" can't surface a structured Result. **Recommendation:** put a non-throwing normalizer in the command path **and** route every dispatcher (including the omitted `InsertEventCommand` and MIDI's `addChordToMeasure`) through one shared planner; do not duplicate per-method checks.

**[LOW] `constants.ts` (file) vs `constants/` (directory) co-exist.** *Verdict: nuance.* Verified both exist; `@/constants` resolves to the file today but is resolution-fragile. Low severity, but the audit's citations of "SMuFL in `src/constants.ts`" are imprecise (SMuFL is in `src/constants/SMuFL.ts`). **Recommendation:** rename the standalone file or fold SMuFL into a `constants/index.ts` barrel; correct the affected-file citations.

**[VALIDATED, strongest] Phase 1D single-source clef geometry.** The infrastructure (`clef.ts` `CLEF_REFERENCES` + `referenceLine`) already exists and is correct; `positioning.ts` keeps a divergent copy (tenor offset 18 vs correct 12) and the inverse `getYToPitch` special-cases only `'bass'`. Deriving one forward formula and **inverting it** for hit-testing is load-bearing, not redundant. Proceed as specified, with the pitch→offset→pitch round-trip regression test.

### Product / scope lens

**[HIGH] Phase ordering: export fidelity (Phase 2) is mis-sequenced ahead of interactive/API correctness (Phases 3–4).** *Verdict: flag.* Verified there is **no import path** (`jsonExporter` is stringify-only; import is "Coming Soon"). For an embeddable *editor*, export is a secondary one-directional interop path, while transpose data-loss, API bar-overfill/selection-orphaning, and minor-key entry hit every interactive session and every integrator of the headline Machine-Addressable API. Phase 2 also gold-plates (`<backup>`, `<part-symbol>`, `<degree>`, per-part keys) an exporter no test parses. **Recommendation:** foundations → structural invariants + transpose → **then** deep export fidelity. Keep only the trivial early export wins (`<alter>`, content-derived `<divisions>`, ABC tie/natural).

**[MEDIUM] Tuplet ×105 migration — FIX-LATER / scope-down.** Do the cheap local fixes (export divisions, `migrateChordTrack` accumulation, tiling invariant test) now; defer the full grid migration until a concrete on-screen bug (not the latent map-key risk) is demonstrated.

### I/O patterns lens

**[HIGH] `getScore()` "stale data" workaround is masking, correctly root-caused.** *Verdict: validated — the verification doc's best I/O conclusion.* Verified `getScore` reads `engine.getState()` synchronously and `dispatch` updates `engine.state` before returning — so there is **no engine staleness**; the only "staleness" is React render timing, which `getScore` bypasses. TESTING.md's stale-state claim is itself stale post-ADR-006 and contradicts ADR-006. Fix: wrap mutations in `act()`, assert on `getScore()` content, delete the TESTING.md selection-proxy guidance, reconcile the ADR/TESTING contradiction.

**[HIGH] Fail-soft without boundary validation masks corruption.** *Verdict: validated.* The smoking gun is self-contradiction within one function: `addNote` capacity at `entry.ts:144` uses the 64 default while the overflow handler at `entry.ts:287-288` reads the real time signature. `loadScore` returns `ok:true` "Score loaded" for structurally invalid input. **Recommendation:** prefer reject-with-actionable-`ok:false` (offending staff/measure + over/under amount) as default; make normalize-on-ingest opt-in (silently mutating a caller's score is itself a POLA violation).

**[HIGH] Export-without-import asymmetry + migration bypass.** *Verdict: validated.* Confirmed: `LoadScoreCommand.execute` returns `newScore` verbatim; `migrateScore` runs only at mount (`useScoreLogic.ts:99`) and isn't even exported from `index.tsx`. Legacy global-quant chordTracks load mis-anchored via the runtime API. **Correction to the audit:** its "malformed loadScore crashes at `useScoreLogic.ts:149`" is **overstated** — verified `ScoreEngine.setState` rejects staves-less state and **no-ops silently** on the runtime path; the crash is only reachable via the mount/`initialScore` path.

**[HIGH] Selection-repair via React `useEffect` is necessary but insufficient.** *Verdict: validated, with caveat.* ADR-006's premise is synchronous mutate-then-read with no React commit between — so a `api.setStaffLayout('single').transposeSelection()` chain dereferences the stale staff *before* any effect runs. **Recommendation:** clamp/clear selection inside the structural commands' atomic transition, in addition to the reactive effect.

**[MEDIUM] `window.riffScore` global-singleton registry.** *Verdict: nuance.* SSR is genuinely safe (verified: `typeof window` guard + effect-gated writes). But `.active` is a "last-mounted/last-writer" pointer, **not** focus-tracking as COOKBOOK claims, and the un-namespaced global is clobberable by a second bundle. **Recommendation:** keep the registry for devtools ergonomics but make the **returned `useScoreAPI` handle** the source of truth; make `.active` focus-driven or rename it; warn on double-bundle version mismatch. Do not remove the global (the E2E seed seam and demo depend on it).

**[MEDIUM] Read-vs-notify coherence hazard (undocumented).** *Verdict: flag — new, not in either doc.* `getScore()`/`operation`/`batch` fire synchronously, but `on('score')`/`on('selection')` fire **asynchronously** via `useEffect` with reference-equality dedup keyed on React state. A programmatic batch can be observed by `getScore()` yet missed/lagged by an `on('score')` subscriber. **Recommendation:** drive `score`/`selection` events from `ScoreEngine.subscribe` (like `batch` already is), and document that `getScore()` — not `on('score')` — confirms a just-issued imperative mutation.

**[MEDIUM] `api.play()` ignores chords.** *Verdict: validated.* Verified `playback.ts` imports only `scheduleTonePlayback` (melody-only) while the UI uses `scheduleScorePlayback`. Adopt fix (b): collapse to one scheduling primitive so no caller can omit chords.

## What the Docs Got Genuinely Right (balance)

- **The catastrophic export bug is correctly the #1 risk:** verified the MusicXML exporter emits only `<step>/<octave>` with no `<alter>` — every sharp/flat exports a semitone wrong into software users actually open. Fix is ~3 lines mirroring the existing root-alter code.
- **Minor keys broken in the core theory path:** verified `Key.majorKey('Em').scale === []`; the staff visibly contradicts its own printed signature. Correct critical, correct low-risk fix (the right resolver already exists).
- **`setAccidental`/`toggleAccidental` are silent no-ops** on a public API that returns `ok:true` — verified; a clean, well-located critical.
- **The transpose undo lossiness** (`TransposeSelectionCommand` re-runs the inverse instead of snapshotting `oldPitch`) is the cleanest, most defensible finding in its cluster — verified the contrast with `ChangePitchCommand`.
- **The dual-source-of-truth, treble/4-4-hardcode, and UI-only-validation root causes** are real, well-evidenced unifying themes that genuinely organize the remediation.
- **The first-principles posture** (reject epsilon tolerance, reject keep-both-and-sync, reject parallel tables; fix models not symptoms) is exactly right for a notation engine where wrongness is silent.
- **The verification doc's flawed-test hunt and weak-assertion/no-oracle diagnosis** are precise and verbatim-verified — `grandStaffAlignment` being `Math.max===Math.max`, the vacuous `frequency>0`, the hedged enharmonic assertion, and the institutionalized `getScore()` staleness workaround are all real and well-targeted.
